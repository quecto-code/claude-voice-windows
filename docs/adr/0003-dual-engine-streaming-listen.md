# listen は vosk ストリーミングと faster-whisper 確定の二段構成にする

`voice_listen` は STT エンジンを 2 つ併用する。録音中は **vosk**（RNN-T 系の真のストリーミング ASR）に音声チャンクを逐次流し、暫定認識（partial）とキーワードスポッティングを担わせる。発話が確定したら、その音声全体を **faster-whisper** に通して高品質な最終テキストを得る。

## なぜ

確定ワード（requirements の Requirement 9）を「喋っている最中に」検出するには、録り終えてから一括変換する方式では不可能で、フレーム単位で partial を出せる逐次推論が要る。Whisper 系はオフライン 30 秒窓モデルで partial をネイティブに出せないため、ストリーミングが本来得意な vosk を確定ワード検出と暫定表示に充て、最終文字起こしの品質は faster-whisper で担保する分業とした。副次的に、喋った内容を画面に暫定表示でき、誤認識にその場で気づける。

## Considered Options

- **沈黙確定一本（vosk なし）**: `sox` の無音検出だけで録音停止。最小構成だが確定ワードは実現できず、リアルタイム表示も出せない。
- **faster-whisper のローリング推論**: 伸びるバッファに対し周期的に再推論して疑似ストリーミング。エンジンは 1 つで済むが GPU 負荷が高く実装も煩雑。
- **二段構成（採用）**: 各エンジンを得意分野に充てる。エンジン 2 つと並行処理の複雑さを引き受ける代わりに、確定ワード・リアルタイム性・最終品質をすべて満たす。

## Consequences

- 沈黙確定（Requirement 8）と確定ワード（Requirement 9）の両方が確定トリガーとして併存する。
- vosk の常時ストリーミング推論が録音中ずっと走る分、CPU/GPU 負荷が増える。
- 認識精度はユーザに見せる暫定表示（vosk）と Claude に渡す最終指示（faster-whisper）で異なりうる。
