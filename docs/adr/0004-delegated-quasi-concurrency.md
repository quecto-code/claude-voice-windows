# 音声ループを委譲型準並行に拡張する（バックグラウンド・サブエージェント）

claude-voice は当初 MCP ツール駆動のターン制（処理中は割り込み不可、[ADR-0001](./0001-mcp-tool-driven-turn-based.md)）を採ったが、重い作業の最中にユーザが沈黙のまま待たされる問題を解くため、**フォアグラウンドの `/voice-chat` 会話ループが重い作業をバックグラウンド・サブエージェント（Claude Code の `run_in_background`）に委譲し、自身は会話を続ける「委譲型準並行」**に拡張する。これは ADR-0001 が「Claude は 1 ターン内で逐次実行されるため処理中に同時に音声を聞くことは原理的にできない」と述べた前提を、ADR-0001 が「実装が重い」と退けた外部プロセス駆動なしに覆すものである。

## Considered Options

- **(A) ターン制を維持（[ADR-0001](./0001-mcp-tool-driven-turn-based.md)）**: 実装は最小だが、重い作業中はユーザが沈黙のまま待たされる問題が残る。
- **(B) 委譲型準並行（採用）**: 会話を止めずに重い作業を裏で進める。走行中ジョブの即時 kill（完全な割り込み）は狙わない＝「準」並行。`run_in_background` サブエージェントで、外部プロセス駆動なしに成立する。
- **(C) 外部プロセス駆動の完全モデル B**: 別プロセスでマイクを常時受け、走行中の処理を kill/ESC で中断する。処理中の真の割り込みが可能になる反面、Claude Code の起動・制御を外側から握る必要があり実装が重くリスクが高い（ADR-0001 で却下済み、本 ADR でも引き続き見送り）。

## Consequences

- **同時に走るバックグラウンドジョブは 1 個まで。** 走行中に新たな重い作業を指示されたら、暗黙に並列起動せず「終わってからにするか／今のを止めて差し替えるか」を音声で確認する。
- **会話状態（ターン・silent カウント・終了コマンド判定・進行中ジョブのハンドル 1 個）はフォアグラウンドの `/voice-chat` が唯一保持する。** MCP サーバはステートレスのまま不変（[ADR-0001](./0001-mcp-tool-driven-turn-based.md) 由来の設計判断を堅持）。本変更で `voice_listen` / `voice_speak` とサーバ実装は変えない。
- **音声を出すのはフォアグラウンドのみ（単一スピーカー）。** バックグラウンドジョブは voice ツールを呼ばず、結果・進捗はテキストで親に返す。
- **完了・進捗の読み上げはターンの切れ目で行う（瞬時割り込みではない）。** フォアグラウンドのターンは依然逐次。これが「準」並行たる所以。
- **進捗報告は走行中の silent タイマ（onset timeout ~30 秒）を転用する。** ユーザの沈黙で `voice_listen` が `silent` で戻るたびに「まだ～中」を読み上げる。粒度が ~30 秒と粗いのは**既知のトレードオフ**で、まずこの形で実装し運用しながら調整する。
- **進行中ジョブがある間は silent 自動終了（requirements の Edge Case 2）を抑止する。** ジョブを待って黙っているユーザを蹴り出さないため。ジョブが無いときだけ従来どおり 2 連続 silent で自動終了。
- **破壊的・不可逆な操作はフォアグラウンドでのみ実行する。** 委譲先のバックグラウンドジョブは破壊的操作を実行せず親に差し戻し、親が音声ブロッキング確認（requirements の A-2 / [ADR-0002](./0002-skip-permissions-with-voice-confirmation.md)）を経てから実行（または許可付きで再委譲）する。準並行では走行中ジョブを音声で即時 kill できないため、「割り込めるから事前確認は任意」というモデル B の論理は採らず、A-2 のブロッキング確認を緩めずに維持する。
- **読み上げの役割が 3 → 4 に増える（進捗報告を追加）。** requirements の「対話モデル」はモデル A／B の二択ではなく、第 3 の「委譲型準並行」を採る。

## 下流への波及（spec-amend が指揮）

本 ADR の確定を起点に、requirements（対話モデル節の改訂＋進捗報告の新 FR）→ design（相互作用）→ `/voice-chat` プロンプト → 新タスクへ波及する。変更履歴は `docs/changes/`（CH-002）に記録する。なお「1 往復ごとのレイテンシ（STT→Claude→TTS の往復）」はサブエージェントでは解消しない別問題であり、本 ADR の対象外（別途 Performance NFR / perf タスクとして計測してから扱う）。
